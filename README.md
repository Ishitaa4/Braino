# 🧠 Braino — Built for Memory, Not Cramming

> A flashcard app that turns any PDF into a smart, spaced-repetition study deck powered by LLaMA 3.3 via Groq.

**Live Demo:** https://flashcard-kappa-snowy.vercel.app/ 
**Built by:** Ishita Chowdhury  

---

## What It Does

You drop in a PDF — lecture notes, a textbook chapter, study material — and Braino generates 15 high-quality flashcards from it in seconds. But the real magic isn't the conversion. It's what happens after.

Braino uses the **SM-2 spaced repetition algorithm** to schedule your reviews. Cards you know well appear less frequently. Cards you struggle with keep coming back. Over time, the app builds a picture of what you've mastered and what still needs work.

---

## Features

- **PDF → Flashcards** — Upload any PDF and get a clean deck of cards instantly, generated by LLaMA 3.3 (via Groq) with a prompt engineered to produce conceptual, teacher-quality cards — not shallow bullet scrapes
- **Spaced Repetition (SM-2)** — Each card has its own ease factor, interval, and repetition count. Rate cards as Again / Hard / Good / Easy after each review. The algorithm adjusts the next review date accordingly
- **Mastery Tracking** — Every deck shows a live mastery bar, total cards, mastered count, and cards due today
- **Session Summary** — After each review session, see your score, rating breakdown (Again / Hard / Good / Easy), and a card-by-card breakdown with your rating for each
- **Deck Management** — Search across all your decks, see progress at a glance, delete decks you no longer need
- **Auto-refresh** — The home page polls every 6 seconds so due counts update automatically after a session

## How the SM-2 Algorithm Works

Each card tracks three values: `repetitions`, `interval` (days until next review), and `easeFactor` (how easy the card is for you, starts at 2.5).

When you rate a card:
- **Again (0) / Hard (1)** → repetitions reset, interval goes back to 1 day
- **Good (2) / Easy (3)** → interval grows (1 → 6 → interval × easeFactor), repetitions increment
- **easeFactor** adjusts up or down based on your rating

A card is marked **mastered** once you've rated it Good or Easy across multiple sessions. The mastery bar on the home page reflects the percentage of mastered cards in each deck.

## Getting Started (Local)

### Prerequisites
- Node.js v18+
- MongoDB Atlas account (or local MongoDB)
- Groq API key (free at console.groq.com)

### 1. Clone the repo

```bash
git clone https://github.com/Ishitaa4/Braino.git
cd Braino
```

### 2. Set up the backend

```bash
cd server
npm install
```

Create a `.env` file in `/server`:

```env
GROQ_API_KEY=your_groq_api_key_here
MONGO_URI=your_mongodb_connection_string_here
PORT=5000
```

Start the server:

```bash
npm start
```

### 3. Set up the frontend

```bash
cd client
npm install
npm run dev
```

App runs at `http://localhost:5173`

## API Reference

| Method | Endpoint | Description |
|--------|----------|-------------|
| POST | `/api/decks/upload` | Upload PDF, generate flashcards |
| GET | `/api/decks` | Get all decks |
| GET | `/api/decks/:deckId/cards` | Get all cards for a deck |
| GET | `/api/decks/:deckId/due` | Get cards due for review |
| PATCH | `/api/decks/cards/:cardId/review` | Submit a rating, update SM-2 fields |
| DELETE | `/api/decks/:deckId` | Delete a deck and all its cards |

## Key Decisions & Tradeoffs

**Groq over OpenAI** — Groq's inference on LLaMA 3.3 70B is significantly faster (under 2 seconds vs 6–8 seconds for GPT-4). For a flashcard app where the upload experience needs to feel snappy, this mattered.

**SM-2 over Leitner boxes** — SM-2 tracks ease per card individually rather than sorting cards into buckets. This gives finer-grained scheduling, especially for a student with a mixed deck of easy and hard concepts.

**No auth in v1** — Skipped user authentication to ship faster. All decks are currently global (anyone using the app sees all decks). This is the most significant tradeoff and the first thing I'd add.

**PDF sliced at 8000 chars** — For the initial version, only the first 8000 characters of a PDF are sent to the model. This keeps latency low and costs minimal but means longer documents get partial coverage. The fix is chunking + multi-pass generation.

---

## What I'd Improve With More Time

- **User authentication** — Per-user decks with login/signup
- **Smarter PDF ingestion** — Chunk longer PDFs and generate cards across all chunks, not just the first 8000 characters
- **Richer card types** — Fill-in-the-blank, multiple choice, image-based cards
- **Streak and motivation layer** — Daily streaks, review reminders, confetti on deck completion
- **Mobile-first design** — The current UI works on mobile but wasn't designed for it
- **Deck editing** — Let users edit, add, or delete individual cards after generation
